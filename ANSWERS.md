




# 1. If this NestJS backend evolved into a production QDC system with thousands of orders and concurrent users, what changes would you make to the data access layer and service design, and why?

The current implementation stores all orders inside an in-memory array within OrdersService. While this approach is simple and works well for a coding assignment, it would not be suitable for a production system because the data would be lost whenever the server restarts.

For a production-ready solution, I would replace the in-memory storage with a database such as PostgreSQL. Using an ORM like Prisma or TypeORM would help manage database operations and relationships between entities such as orders, garments, customers, and payments.

I would also separate business logic from data access by introducing repositories. This would keep services focused on business rules while repositories handle database communication. For larger workloads, I would introduce pagination, caching using Redis, and database indexing to improve performance.

These changes would make the application scalable, reliable, and capable of handling thousands of concurrent users.



# 2. The GET /api/orders/:id endpoint currently returns either an Order or an { error: string } object. What are the tradeoffs of this approach, and how would you improve error handling and API design for a real-world POS/ordering API?

Returning either an Order object or an error object creates inconsistent response structures. The frontend must inspect the response manually to determine whether the request succeeded or failed.

In a production application, I would use proper HTTP status codes and NestJS exception handling. For example, if an order does not exist, the API should return a 404 Not Found response using NestJS's NotFoundException.

This makes the API more predictable and easier to consume because clients can rely on status codes rather than inspecting object structures. I would also implement a global exception filter so that all API errors follow a consistent format across the application.



# 3. The React frontend calls the backend directly with fetch('http://localhost:3001/api/orders') inside useEffect. If this UI needed to grow into a more complex dashboard (filters, pagination, multiple API calls), how would you structure the frontend code differently to keep it maintainable?

As applications grow, placing API calls directly inside components can make the code difficult to maintain. UI logic and networking logic become tightly coupled.

To improve maintainability, I would move API communication into a dedicated service layer. For example, a file such as client/src/services/ordersApi.ts could contain all API-related functions.

For state management and caching, I would use React Query. It provides loading states, caching, automatic refetching, and error handling. Components would focus only on rendering data while the service layer manages communication with the backend.

This separation of concerns would make the frontend easier to scale and maintain as additional features are introduced.



# 4. Orders in QDC involve multiple business workflows (garment processing, delivery, billing, prepaid packages). Given the current Order and Garment types in OrdersService, what potential edge cases or missing fields do you see, and how would you evolve the domain model to better support real-world laundry operations?

The current model is intentionally simple and focuses only on customer names, garments, and garment statuses. However, real-world laundry operations require significantly more information.

Some important missing fields include customer contact details, payment status, order pricing, delivery information, pickup schedules, garment quantities, and special cleaning instructions.

I would also introduce status history tracking instead of storing only the current status. This would allow the system to record when garments moved from one stage to another and provide operational visibility.

Over time, I would separate the domain into dedicated entities such as Customers, Orders, Garments, Payments, Deliveries, and Staff Assignments. This would make the system easier to extend and support more complex workflows.



# 5. Imagine the OrdersService methods were initially generated by an AI tool and then lightly edited. What specific risks do you see in relying on AI-generated code in this backend and frontend, and what review/debugging practices would you apply before shipping features to production?

AI-generated code can improve development speed, but it should never be accepted without review. Generated code may contain hidden bugs, incorrect assumptions, security vulnerabilities, or inefficient implementations.

One example from this assignment is filtering logic. The implementation may appear correct initially but fail under certain edge cases if not tested properly.

Before deploying AI-generated code, I would perform manual code reviews, unit testing, integration testing, TypeScript validation, linting, and edge-case testing. I would also verify that the implementation follows project architecture standards and does not introduce duplicated logic.

AI should be treated as a productivity tool rather than a replacement for engineering judgement.



# 6. Suppose the QDC dashboard needs to show a real-time view of garments moving through statuses (e.g., a board that updates when a garment becomes ready). With the current REST endpoints and architecture, how would you approach adding near real-time updates, and what tradeoffs would you consider?

The current architecture relies on REST endpoints, which means the frontend only receives updates when it explicitly requests them.

To support near real-time updates, I would introduce WebSockets using NestJS Gateways. Whenever a garment changes status, the backend could emit an event that connected clients receive instantly.

An alternative approach would be periodic polling, where the frontend requests updates every few seconds. Polling is simpler to implement but generates unnecessary network traffic and introduces delays.

For a production dashboard where operational visibility is important, I would prefer WebSockets because they provide immediate updates and a better user experience. The tradeoff is increased implementation complexity and connection management, but the benefits outweigh the costs for a real-time operations dashboard.
